今天这期，讲一篇 Lance Martin 在 X 上转发的内容，标题叫《Launching Claude Managed Agents》。

这次还是按我们现在默认的做法：
不把它硬压成几句话，而是尽量保留原文想表达的信息，只做整理和口语化处理，让它适合连续收听。

先说这篇东西到底在讲什么。
它讲的不是单纯又发了一个新模型，也不是再给 Claude API 加一个普通参数。
它讲的是一个更高一层的东西：Anthropic 把“长任务 agent 的运行环境”，开始以托管服务的方式抽出来了。
名字就叫 Claude Managed Agents。

Lance 自己在 X 上的概括很直接。
他说，这是 Claude API 里的一个新东西。
你提供 agent configuration，也就是 agent 的配置，再给它一个任务；剩下的 agent harness 和托管基础设施，由 Anthropic 来处理，目的是让它在长任务周期里也能稳定工作。

这句话其实已经把核心价值说出来了。
过去很多人做 agent，最头疼的问题并不是“模型能不能答对一道题”，而是：
一个任务如果持续很久，中间要调用很多工具，要跨多个步骤，要处理失败重试、上下文管理、状态保存、成本和可靠性，那你到底怎么把整个系统搭起来。

也就是说，真正难的往往不是那一下推理，而是 agent harness，也就是包住模型、工具、状态、失败恢复和执行策略的那一层系统。
Anthropic 这次相当于在说：
这一层，不用你全自己搭了，我们开始把它做成 Claude API 的一部分。

从这篇内容和 Lance 同一串帖子的补充看，Managed Agents 的重点至少有几层。

第一层，是它本质上是“hosted agent in the Claude API”。
也就是说，这不是本地跑一个脚本、自己写个循环、自己接一堆工具那么简单，而是 Anthropic 在云端提供一种长期运行的 agent 服务。
用户要做的，不是亲自盯着每一个 step，而是定义好 agent 的配置和任务，然后把运行和可靠性交给平台。

第二层，是它针对的是 long task horizons，也就是长时长任务。
这很重要。
很多 demo 里的 agent，看起来好像很强，因为任务只有几轮交互、几次工具调用，甚至几十秒就结束了。
但真实世界里，一旦任务变长，问题就会开始出现：
上下文会变长，状态会漂移，工具调用会失败，某些中间结果需要重试，某些步骤可能需要等待外部资源，甚至整条链路跨越很长时间。
所以 Anthropic 这次真正想解决的，不是“一个 agent 能不能完成短任务”，而是“agent 能不能在长任务里可靠地活下来”。

第三层，是 harness 本身会过时。
Lance 之前就写过一个很关键的判断：
agent harness 往往编码了很多“模型做不到什么”的假设，但这些假设会随着 Claude 能力变强而逐渐过时。

这个判断很值得展开。
因为很多人做 agent 框架时，都会在外面套很多控制逻辑：
什么时候拆步骤，什么时候截断上下文，什么时候总结历史，什么时候禁止某种行为，什么时候强制走某个流程。
这些做法在某个时间点可能是对的，因为模型还不够强。
但随着模型能力提升，原来为了“弥补模型不足”而设计的大量壳层，反而会变成束缚，甚至会让系统更脆弱、更难维护。

所以你会看到 Lance 在另一条帖子里说得很明确：
这里有一个张力。
一方面，你想给 Claude 更通用的工具，让它自己管理自己的动作和上下文；
另一方面，你又要用 harness 去设置边界，比如安全边界、产品边界、用户体验边界。
这两件事并不完全一致。
Managed Agents 可以理解为 Anthropic 开始正式接手这层张力，试图在平台层面把它做成一个更可靠的系统。

第四层，是“memory”这件事，在这里并不是外界容易误解的那种 memory block。
Lance 还专门在跟别人讨论时澄清了一点：
这里不是在用某种 memory blocks；更准确地说，是 `memory_store` 给 agent 一个 filesystem，而他们很多工作其实都在围绕“如何给 Claude 一个文件系统来做记忆”展开。

这点很关键。
因为很多人一听到 agent memory，就会想到向量数据库、长期记忆模块、profile、episodic memory 之类更抽象的结构。
但 Anthropic 这一拨看起来更重视一个很朴素、但工程上非常强的东西：
给模型一个文件系统，让它像一个实际工作的程序一样，去写文件、读文件、组织状态、保存中间产物。

这个思路的含义是：
他们不只是把“memory”当成一个检索问题，而是把它当成一个操作环境问题。
也就是，真正长期运行的 agent，需要的不只是 recall，而是一个可以持续落地工作痕迹的环境。

第五层，是 onboarding 和生态入口。
Lance 还提到，一个很好上手的方式，是直接使用他们做进 Claude Code 的 skill。
做法大概是：
先更新 Claude Code，
然后进入 Claude Code，
再运行 `/claude-api managed-agents-onboarding` 这样的子命令。

这说明 Anthropic 不只是把 Managed Agents 做成一个底层 API，然后要求开发者自己啃文档。
它也在试图通过 Claude Code 这个现成入口，把托管 agent 能力直接交给开发者体验。
也就是说，这既是 API 产品，也是开发者工作流产品。

如果把这篇内容放进更大的行业背景里，它其实在回答一个更大的问题：
为什么现在越来越多的人，开始把“agent 平台”看成比“模型本身”更难做的部分？

原因就在这。
模型只是大脑的一部分。
真正进入生产以后，你还需要：
任务编排，工具调用，状态管理，失败恢复，权限隔离，执行环境，日志，可观测性，成本控制，以及长任务里的可靠性保障。
这些东西加在一起，才构成一个能持续运行的 agent 系统。

所以从产品定义上说，Claude Managed Agents 更像是在卖一套“agent 运行时和基础设施托管服务”，而不只是卖一次推理请求。
换句话说，Anthropic 想往上吃一层价值。
过去开发者调用模型 API，自己搭 harness。
现在 Anthropic 开始说：
如果你的目标不是单次调用，而是长期运行、可恢复、可管理的 agent，那么这层也可以由我们来托管。

这对开发者意味着什么？
我觉得可以拆成几种不同影响。

第一，对很多中小团队来说，这会显著降低他们做 agent 产品的门槛。
因为真正难的往往不是 prompt，而是系统工程。
如果平台把长任务管理、基础设施和可靠性托起来，很多团队就可以把精力从“怎么搭一套能跑的壳”转向“到底想让 agent 干什么”。

第二，它也会让一部分独立 agent framework 面临更明确的竞争。
因为如果平台本身开始托管 harness，很多原本由第三方框架提供的价值，就会被上游平台收走。
特别是那些主要价值集中在任务循环、状态管理和基础设施封装上的框架，会更明显感受到压力。

第三，它也会改变大家对“agent product”边界的理解。
以前很多人把 agent 理解成：模型加工具。
但这次 Anthropic 给出的方向更像是：
agent 其实是“模型 + 工具 + 文件系统 + 长任务运行环境 + 托管恢复机制 + 平台级可靠性”的组合体。
这个定义明显更接近生产系统，而不是 demo。

当然，这里面也有很现实的问题。
比如，有人已经在评论里指出，如果不能把 chat history 或 agent state 更方便地传进 SDK 里做长期持久化，那在生产环境里仍然会有很多限制。
还有人会关心，平台托管越多，是不是就意味着系统更耦合到某个特定提供方的运行方式。
以及，如果 memory 主要建立在文件系统抽象上，它对不同类型任务的泛化到底有多强。
这些问题目前都还不能说已经彻底解决。

但即便如此，这次发布依然很值得重视。
因为它说明 Anthropic 已经明确做了一个判断：
下一阶段竞争，不只是模型分数和上下文长度，而是谁能把“长时长 agent 的运行系统”做成一个真正可用、可托管、可进入开发者工作流的服务。

如果你熟悉我们之前一直在讨论的那个核心问题——也就是 agent 的瓶颈到底在模型、还是在 harness——那这次发布可以看成 Anthropic 给出的一次非常明确的产品答案。
它没有说 harness 不重要。
恰恰相反，它等于是在说：
harness 太重要了，重要到应该由平台亲自下场，把它产品化、托管化、服务化。

最后做个收束。
这篇《Launching Claude Managed Agents》最值得记住的，不是一个新名字，而是三个判断。

第一，真正难的 agent，不是短任务 demo，而是长任务可靠运行。
第二，随着模型变强，很多旧 harness 的假设会迅速过时，所以平台需要重新设计 agent 的运行边界。
第三，Anthropic 正在把 agent 的竞争重心，从单次模型调用，推向“托管 agent 运行时和基础设施”这一层。

所以如果你今天听完只记一句话，我觉得可以记成：
Claude Managed Agents，不只是一个新 API 功能，而是 Anthropic 在正式把“agent harness”这件事，升格为平台产品。
