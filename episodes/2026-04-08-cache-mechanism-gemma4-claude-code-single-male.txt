这一期我们围绕一篇很有代表性的主题来讲，题目叫“搞懂缓存机制，从 Gemma4 到 Claude Code 省 80% Token”。我这里先说明一下，这一期不是逐句朗读原文，而是围绕这篇文章想讲的核心问题，做一版适合声音收听的技术讲解。重点不是复述标题，而是把所谓缓存机制，到底在大模型推理和 coding agent 场景里是什么意思，为什么它能显著省 token，讲清楚。

先从一个最直接的问题开始。为什么很多人一边用大模型，一边又会对 token 消耗特别敏感？因为对现在的大模型系统来说，token 不只是费用问题，它同时还是延迟问题、吞吐问题、上下文长度问题，甚至还是产品体验问题。你如果每一轮都把一大段重复上下文重新塞给模型，那你既会多花钱，也会让响应变慢，还会挤占真正有价值的新信息。所以一旦开始做工程系统，而不是偶尔对话，就迟早会碰到缓存。

但缓存这个词在大模型里经常被讲得很混。有人说的是 prompt caching，有人说的是 KV cache，有人说的是 conversation memory，还有人说的是 agent 自己维护的摘要和状态。表面上都叫缓存，但它们不是同一个层次。要讲清楚这个主题，第一件事就是把层次拆开。

第一层，是最偏推理底层的缓存，也就是大家常说的 KV cache。大模型在生成过程中，并不是每次都从零开始理解整个上下文。对于 Transformer 结构来说，前面已经算过的 token，会形成一组 key 和 value 表示，后面继续生成时，可以复用前面的计算结果，而不是每次重新把全部历史再算一遍。这个机制本质上是为了提升推理效率。你可以把它理解成，前文已经编码好的中间结果被保留下来，后续 token 只需要在这份上下文基础上继续往下走。这个缓存越能稳定复用，生成越快。

但 KV cache 主要发生在一次推理会话内部，或者至少是在同一次请求的连续生成过程中。它能减少重复计算，却不一定直接等于“跨请求省 token”。如果换个说法，就是 KV cache 更偏算力复用，不完全等于账单层面的输入复用。很多人把这两件事混在一起，其实会误判系统设计重点。

第二层，是更接近产品和 API 计费层的 prompt caching。这个层面的意思是，如果你一大段前缀提示词、系统提示词、项目上下文、仓库说明，长期不变，那每次请求都把这些内容完整重新处理一遍，其实非常浪费。于是很多系统会尝试把这一大段稳定前缀做成可复用块。后续请求如果命中了相同前缀，就不必把整段都重新按原成本计算。这里节省的，往往更容易直接体现为 token 成本下降和延迟下降。

这也是为什么在 coding assistant 和 agent 场景里，缓存特别重要。因为 coding 场景最典型的问题就是：有大量稳定但很长的上下文。比如仓库结构、系统提示、工具说明、项目文档、依赖版本、长期规范、上轮中间状态。这些内容有些轮次会变，但不是每一轮都变。如果每次都把它们原样重新喂进去，那成本会迅速膨胀。

所以你再看“Claude Code 省 80% Token”这种表达，真正有价值的不是那个百分比本身，而是它在提醒一个工程事实：在 coding agent 里，大量 token 消耗其实来自重复上下文，而不是来自每一轮真正新增的任务信息。换句话说，如果系统足够聪明，它应该把“恒定背景”和“本轮新问题”分开对待。缓存，就是在做这个分层。

接下来讲第三层，也就是 agent 自己维护的状态缓存。这一层不一定是模型底层或 API 官方给的能力，而是产品或应用自己做的。比如一个 coding agent 可能会把项目结构做成索引，把最近修改的文件列表做成状态，把某些长文档压成摘要，把已经确认过的环境信息独立存下来。这样下一轮不一定需要把原文全文再送给模型，而是只把新的增量部分和必要摘要送进去。这类机制严格说不全是模型缓存，但在实际工程里，它起到的效果和缓存很像：减少重复输入，减少不必要的 token 消耗。

你会发现，一旦从工程视角看，大模型缓存就不再只是一个底层推理优化技巧，而变成了一个系统架构问题。你必须决定：哪些信息是稳定前缀，哪些信息是动态上下文，哪些信息应该做摘要，哪些信息应该做索引，哪些信息必须原文保留。这些判断，会直接影响成本和体验。

那为什么这个问题会同时从 Gemma4 讲到 Claude Code？因为它反映的是一个跨模型、跨产品都成立的共同现象。无论是开源模型、商用模型、聊天产品还是 coding agent，只要你开始长期、多轮、带项目上下文地使用模型，你就会遇到同一个瓶颈：重复输入太多。模型越强、上下文越长、任务越复杂，这个问题越突出。也就是说，缓存不是某个模型的小技巧，而是大模型产品进入“真正可用工程阶段”后必须正视的基础设施。

再进一步说，缓存之所以重要，不只是因为它能“省钱”，还因为它改变了你构建应用的方式。如果没有缓存，你就会倾向于尽量缩短提示词，把很多上下文都砍掉，因为每一轮都很贵。这样做会损害能力边界。可一旦缓存做得好，你就更有底气维持一个比较稳定而丰富的前缀：比如把项目规范、协作规则、代码风格、工具说明都保留下来。于是应用的长期一致性会变强。这一点对 coding agent 尤其明显。

举一个很典型的例子。假设你在一个大型代码仓库里工作。你每轮都需要模型知道这些信息：仓库结构、测试命令、技术栈、不能改的模块、代码风格、部署方式、CI 约束，还有最近几轮已经做了什么。如果没有缓存，这一大段背景会非常贵，于是很多系统会偷懒，只给一小部分。结果就是模型这轮知道一半，那轮忘一半，行为不稳定。反过来，如果这些内容被很好地缓存和复用，模型就能在更一致的上下文里持续工作。这样你看到的就不是“偶尔灵光一闪”，而是更稳定的工程表现。

从这个角度看，缓存其实是在买稳定性。你表面上看到的是 token 降低，背后得到的是可持续、多轮、低摩擦的工作流。这个收益往往比账单上的数字更大。

那缓存有没有代价？当然有。第一个代价就是命中条件。如果缓存依赖前缀完全一致，那你系统提示词稍微变一下、文档顺序稍微变一下、前面多一个空格少一个空格，命中率都可能下降。也就是说，想让缓存真正有价值，前端和编排层必须尽量稳定。提示词模板不能一会儿一个样，系统上下文不能每次重排。否则理论上有缓存，实际上老是 miss。

第二个代价是状态管理复杂度。你一旦开始做多层缓存，就要问：缓存什么时候失效？项目文档更新了怎么办？用户切换任务了怎么办？模型版本变了怎么办？工具说明改了怎么办？这些都是缓存一致性问题。传统软件里，缓存失效本来就是最经典的难题之一；到了大模型系统里，这个问题并没有消失，只是换了一种形式出现。

第三个代价是可解释性。缓存做得越深，系统越容易出现一种现象：外部看起来像是“模型记住了”，但实际上是前缀复用、摘要复用、状态索引、KV cache 共同作用的结果。对于终端用户来说，这可能很好；但对于开发者来说，如果没有把缓存边界设计清楚，调试会非常痛苦。你会搞不清到底是模型推理的问题，还是缓存命中了错误版本上下文，还是摘要没有更新，还是某个状态槽已经过期。

所以如果你真的想“搞懂缓存机制”，不是只知道一句“缓存能省 token”就够了，而是要理解缓存到底在系统哪个层次发生。是推理层？是 API 计费层？是 agent 状态层？还是应用自己做的上下文工程？这几个层次分不清，后面一讨论效率优化，就很容易鸡同鸭讲。

回到 coding agent 场景，再讲得更实一点。为什么像 Claude Code 这种产品会特别强调缓存收益？因为 coding agent 的典型输入非常长，而且高度重复。每次运行都要带工具协议、环境描述、项目背景、最近变更、任务目标，很多时候还有文件片段。与此同时，真正本轮新增的信息，可能只是“把这个 bug 修掉”或者“帮我改一个函数”。也就是说，新增信息很短，背景信息很长。如果不缓存，系统绝大多数 token 都花在重复背景上。这样一来，哪怕模型本身已经很强，产品也会因为成本和延迟变得不够顺手。

所以一个好的 coding agent，绝不是只拼模型能力，它还必须在上下文重用上做得足够聪明。具体做法可能包括：稳定化系统前缀，把常驻上下文独立成缓存块，把项目索引和摘要独立存放，把真正需要重新送入模型的内容压到最小。这些动作加在一起，才可能带来标题里那种夸张但合理的节省比例。

那对普通开发者来说，这个主题的最大启发是什么？我觉得是两点。第一，开始把大模型应用看成系统工程，而不是只看提示词技巧。你得关心上下文分层、缓存命中、状态持久化、摘要策略这些更靠近架构的问题。第二，不要以为“token 成本高”只能靠缩短提示词解决。很多时候，更有效的方法不是删信息，而是把重复信息固定下来，让系统有机会复用。

再说一个更长远的趋势。随着模型上下文窗口越来越大，很多人直觉会觉得：上下文都这么大了，缓存是不是没那么重要了？其实恰恰相反。上下文越大，重复信息能占据的体积也越大。如果你有一百万 token 窗口，不代表你就应该每轮都重新塞一百万 token。窗口变大只是上限变高，不代表重复成本消失。甚至因为系统可以吞下更多背景，缓存与上下文工程的重要性会进一步上升。

最后收一下这一期。所谓“从 Gemma4 到 Claude Code 省 80% Token”，如果你把它当成一条营销口号，那听过就算了；但如果把它当成一个工程信号，它其实在提醒你，大模型系统真正进入可用阶段之后，优化重点已经不只是模型本身，而是模型周围的上下文组织方式。缓存不是配角，它正在成为主角之一。谁能把稳定前缀、动态上下文、状态摘要、推理复用这几层关系做清楚，谁就更有可能做出又快、又稳、又省的实际产品。

所以这一期的一句话总结是：大模型缓存的真正价值，不只是让一次调用更便宜，而是让一个长期、多轮、项目级工作流变得可持续。明白这一点，再去看 Gemma4、Claude Code、各类 coding agent 和本地推理框架，你会更容易看出哪些优化是表面文章，哪些优化是架构层真正有价值的东西。
